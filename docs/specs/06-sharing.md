# 06. Шеринг записи победы

## Пласт 1 — запись и сборка

> Зона: `[AGENT]` (движок записи).

- **Цель:** видео победы готово к шеру/скачиванию за ~3 сек после боя.
- **Запись:** без обмена записями между клиентами. Каждый пишет СВОЮ камеру; камеру оппонента берёт из WebRTC-потока (сетевое качество — ок, честно и симметрично). Обмен полными записями отвергнут: десятки МБ, задержка убивает момент.
- **Сборка — БЕЗ рендеринга (ключевое):** бой+эдит рисуются в canvas, MediaRecorder пишет его в реальном времени (canvas.captureStream). Показ эдита = его же запись. Эдит закончился → файл УЖЕ готов (blob в памяти). Рендерить нечего → «3 сек до шера» достижимы.
- **Пишем ТОЛЬКО победную часть** (2 сек до моргания + эдит), НЕ весь бой — иначе кодирование просадит FPS детекции.
- **Источник «2 сек до моргания»** — кольцевой буфер из anti-cheat (см. `03-anti-cheat.md`): лёгкий буфер кадров крутится весь бой, тяжёлое КОДИРОВАНИЕ в файл — только на финале.
- **Формат:** WebM на лету + быстрая серверная конвертация WebM→MP4 для соцсетей (переупаковка контейнера, не рендер, секунды).
- **Разрешение на запись НЕ требуется** (пишем свой canvas, не экран устройства).
- Не забыть: подмешать аудио-трек эдита (canvas звука не содержит, нужен AudioContext); протестить iOS Safari рано (капризен с MediaRecorder).

## Пласт 2 — приватность лица

> Зона: `[AGENT]` (галки при регистрации — логика). Юридическая часть — на владельце.

- **Блюра НЕТ** — замыленный оппонент убивает суть продукта (смысл гляделок — видеть лицо соперника в момент проигрыша). Лицо оппонента в шер-видео остаётся открытым.
- **Барьер** — три галки при регистрации (см. `01-user-flow.md`).
- **ОСОЗНАННЫЙ ЮРИДИЧЕСКИЙ ДОЛГ (принят сознательно):** галки закрывают форму согласия, но НЕ покрывают: (1) контроль оппонента над конкретной публикацией; (2) отзывность с уже выложенного; (3) «18+» декларативна, не проверена → риск с несовершеннолетними; (4) юрисдикции с неотчуждаемым правом на изображение (FR/DE) + BIPA/GDPR при выходе на US/EU.
- **Путь на будущее (НЕ в MVP):** опциональный блюр, если риск станет реальным. Дверь не закрыта.

## Пласт 3 — хранение

> Зона: `[AGENT]`.

- **Где:** MinIO (self-hosted, S3-совместимый) на старте. Переезд на облачный S3 позже — почти безболезненный (тот же API, меняется endpoint + креды).
- **Что храним:** 10 последних матчей в формате победной части (лёгкие клипы). Победитель — свой эдит; проигравший — свою версию (эдит со стороны проигравшего).
- **Ротация:** FIFO — новый момент вытесняет самый старый. Опц. авто-удаление видео старше N дней.
- **Просмотр:** отдельный скрин в профиле (просмотр / скачивание / повторный шер).
- **Нюанс:** «10 последних» = 10 последних СОХРАНЁННЫХ моментов (видео), НЕ полная история всех боёв. Текстовая история боёв — отдельная таблица в БД.
- **⚠ MinIO:** свой сервер → нужны бэкапы и слежение за местом.

## Технология эдитов

> Зона: движок проигрывания/подстановки видео — `[AGENT-SCAFFOLD]` (слот-плейсхолдер); сами анимации эдитов — `[OWNER-DESIGN]`; слияние — `[MERGE]`.

- Эдиты — **готовые HTML/Canvas/WebGL шаблоны**, произведённые в Claude Design (его выходной формат — HTML с анимацией внутри: CSS / JS timeline / Canvas 2D / WebGL).
- В рантайме: шаблон рендерится поверх живого видео в canvas, пишется на лету. НЕ генерация в рантайме — Claude Design это инструмент производства шаблонов заранее.
- Тяжёлые эдиты (WebGL) — ОК, т.к. играют ПОСЛЕ боя (детекция уже не крутится, FPS свободен). Лёгкий фолбэк (CSS/Canvas 2D) для слабых устройств.
- 10 бесплатных + 10 платных эдитов (см. `08-monetization.md`).

### Что делает агент vs владелец (эдиты)

- `[AGENT-SCAFFOLD]`: движок, который берёт шаблон эдита, подставляет в него живое видео из canvas, проигрывает ~10 сек, синхронит трек, пишет через MediaRecorder. На время разработки эдит = прямоугольник-слот, играющий 10 сек.
- `[OWNER-DESIGN]`: сами HTML/Canvas/WebGL анимации (делаются в Claude Design).
- `[MERGE]`: вмерж готовых анимаций в движок (таска создаётся после того, как owner отдал ассеты).
