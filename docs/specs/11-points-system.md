# 11. Система поинтов (вовлечение + retention + крипто-аирдроп)

> Зона: механика поинтов (баланс, леджер, начисление, трата, холд) — `[AGENT]`. UI-виджет поинтов + инфо-панель — `[AGENT-SCAFFOLD]` каркас, финальный вид — `[OWNER-DESIGN]`. Иконка/визуал поинтов — `[OWNER-DESIGN]`.
>
> **Важно для feeder:** в этом разделе строго разделены **[MVP — задачи флота]** и **[Человеческий пререквизит]** (токен, аирдроп-распределение, юр-часть, анти-абуз). Флот строит ТОЛЬКО первое. Второе помечать `Manual prerequisite (human): …`, задачи не создавать.

**Цель:** повысить вовлечение и процент возврата; ещё одно направление удержания. Поинты — soft-валюта, которую можно ТОЛЬКО заработать активностью (не купить за деньги).

---

## [MVP — задачи флота]

Полностью реализуемо в MVP. Токен на этом этапе НЕ существует — поинты пока чисто игровая валюта + накопительный счётчик.

### Данные (зона `[AGENT]`)

- `points_balance` на пользователя (источник истины — Postgres).
- `points_ledger` — **append-only** леджер транзакций: `user_id`, `delta` (+/−), `reason` (enum: `match_win`, `level_up`, `shop_spend`; enum расширяемый под будущие активности), `ref_id` (матч/покупка), `created_at`. Баланс = сумма леджера (или денормализованный счётчик, обновляемый в той же транзакции).
- Денежные/многошаговые операции — в явной транзакции (канон `go-backend-conventions`). Списание поинтов в магазине атомарно с выдачей айтема в инвентарь.
- Redis — кэш баланса для быстрых чтений (источник истины остаётся Postgres).
- Где живёт код (своя `services/points/` или часть магазин/экономика-сервиса) — **архитектурное решение PM-агента** по `go-backend-conventions`; спека это не фиксирует.

### Начисление (earning) — только бесплатно, за активность

- **За победу в матче — ТОЛЬКО победителю.** Проигравший за матч поинты НЕ получает (осознанное решение: за участие не даём).
- **За ап уровня** в рейтинговых матчах.
- Система расширяемая: новые активности по мере добавления → новые значения `reason` в леджере.
- **Поинты-за-пост эдита в соцсети — ОТМЕНЕНЫ в MVP.** Автоматически верифицировать публикацию нельзя (нет колбэка от TikTok/Instagram; на вебе Web Share API не возвращает результат; авто-хештег юзер может стереть). Проверенный шеринг (TikTok OAuth + поллинг по вотермарку) — пост-MVP.
- **Числа начисления — config-driven с плейсхолдерами.** Конкретные значения (сколько поинтов за победу/ап) НЕ фиксированы (экономика TBD). Реализовать как конфиг-константы с временными значениями (`config.yaml`), НЕ хардкод — финальные числа подставит владелец. Это не повод для `needs-human`: критерий ясен, значение параметризуется.

### Трата (spending) — ингейм-магазин

- За поинты покупаются: отвлекалки + базовые эдиты (и любые будущие айтемы).
- **Каждый айтем магазина несёт ДВЕ цены:** `money_price` И `points_price` (nullable). `points_price = null` → айтем только за деньги.
- **Деление премиум (только деньги) / базовый (можно за поинты)** — TBD, определится с конкретным списком эдитов/отвлекалок. До тех пор: схема поддерживает обе цены, конкретная разметка айтемов — позже (владелец/экономика).
- Списание поинтов атомарно с выдачей в инвентарь (см. Данные).

### Холд (hold)

- В MVP холд — это просто **отсутствие траты**: «захолдленный объём» = текущий баланс. Отдельной механики блокировки НЕ требуется.
- UI отображает баланс как «накоплено» — это и есть будущая база для аирдропа.

### UI (зона `[AGENT-SCAFFOLD]` каркас → `[OWNER-DESIGN]` финал)

- Поинты на домашнем экране (после авторизации), **справа сверху**: число + иконка.
- Клик → инфо-панель: (1) за что начисляются (перечень активностей, **БЕЗ конкретных чисел**); (2) куда тратить — ингейм-магазин + «можно холдить на будущий аирдроп» + обтекаемое объяснение принципа распределения (доля пула пропорционально холду, **без гарантий и без цифр**).
- `[AGENT-SCAFFOLD]`: виджет (число + плейсхолдер-иконка) и инфо-панель со статичным текстом на нейтральном дизайне. `[OWNER-DESIGN]`: финальная иконка/визуал и стиль панели.
- **Текст инфо-панели сверяется с юр-формулировками (ниже)** — без обещаний денежной ценности.

## [Человеческий пререквизит] — НЕ задачи флота

Feeder: помечать `Manual prerequisite (human): …`, задачи не создавать. Флот это НЕ строит.

### ⚠⚠ Крипто-токен и аирдроп — самый серьёзный риск проекта

Обещание будущего токена за холд — НЕ техдолг, а потенциальный **регуляторный риск** (вплоть до уголовного в части юрисдикций). Требует юриста ДО запуска токен-части, а не «после MVP».

- **Суть:** тест Хауи (США, SEC) — обещание токена с ожиданием прибыли от усилий платформы может быть квалифицировано как незарегистрированное предложение ценных бумаг. ЕС (MiCA) — публичное предложение криптоактивов требует регистрации/раскрытий. Сторы (Apple/Google) — жёсткие правила про крипто-обещания.
- **Аудитория международная без геоблока США = повышенный риск (принято сознательно).** Notcoin/Blum по публичным паттернам обычно ИСКЛЮЧАЮТ резидентов США из аирдропа (геоблок/KYC) — вероятно, их главный защитный приём. Здесь геоблока нет → риск выше.
- **Рычаги снижения (на будущее):** (1) отдельное юрлицо-эмитент (foundation) в крипто-дружественной юрисдикции (BVI / Каймановы о-ва / ОАЭ / Сингапур / Швейцария); (2) геоблок США (оценить ЕС/UK); (3) токен как utility, не как инвестиция; (4) ToS ЯВНО без гарантий: «поинты не имеют денежной ценности, не являются ценной бумагой, не гарантируют никаких токенов; любой аирдроп — на усмотрение и не гарантирован» — буфер между маркетингом и обязывающими документами; (5) поздняя токенизация.
- **Обязательно:** юрист по securities/крипто до фактического выпуска токена.

### ⚠ Анти-абуз начисления — НЕ в MVP, но ОБЯЗАТЕЛЕН перед запуском токена

- Пока токена нет — фармить нечего (накрутка поинтов ради копеечного айтема в магазине неопасна).
- Как только поинты конвертируются в токен с денежной стоимостью, они становятся прямым путём к деньгам → мультиаккаунты, договорные победы между своими аккаунтами, боты 24/7 ради доли аирдропа. Без защиты аирдроп раздаст токены ферме ботов, а не реальным игрокам (бьёт по справедливости, ценности токена, репутации).
- Защиту (anti-multiaccount, детект договорных матчей, лимиты/аномалии начисления) ставить **вместе с юристом, перед токеном**. В MVP — не строим.

## Зоны ответственности

- `[AGENT]` — баланс, append-only леджер, начисление (победа/ап), списание в магазине (атомарно с инвентарём), кэш баланса в Redis, двойная цена айтемов в схеме магазина.
- `[AGENT-SCAFFOLD]` — виджет поинтов (число + плейсхолдер-иконка) и инфо-панель со статичным текстом на нейтральном дизайне.
- `[OWNER-DESIGN]` — финальная иконка/визуал поинтов, стиль инфо-панели.
- `[MERGE]` — вмерж финальной иконки/стиля в каркас виджета.
