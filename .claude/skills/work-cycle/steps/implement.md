Implement issue #N (already claimed by you — you are the assignee). Print `[IMPLEMENT] #<N> start`.

> Board status is automated by the project's GitHub workflows (issue assigned → In Progress, PR merged → Done). Do NOT touch the Projects board from here — claiming the issue (the assignment done in `select.md`) is what moves the card.

GitHub issue/PR operations use the **GitHub MCP** tools (`mcp__github__*`, owner `pizdagladki`, repo `full`); local code + git via Bash.

1. `mcp__github__get_issue` (#N). If the acceptance criteria are unverifiable/unclear → apply `needs-human` (`mcp__github__update_issue`), exit. Do NOT guess.
2. **Sync first:** `git fetch origin`. Work in a separate git worktree, branching `feat/N-<slug>` off the freshly-fetched `origin/main` (NOT a stale local `main`): `git worktree add ../<slug> -b feat/N-<slug> origin/main`.
3. Plan (brain): for a non-trivial task enter plan mode and draft a plan (WITHOUT human approval); for a trivial task a one-line plan is fine. You do NOT write code yourself — the `coder` subagent does (next steps).
4. **Delegate the implementation to the `coder` subagent** (Sonnet): pass it the absolute worktree path, the zone from the issue's "Service / area" section, the acceptance criteria, and your plan. It codes strictly within that zone per `go-backend-conventions` (+ `new-service`/`new-resource`; `frontend-conventions` for a `frontend` zone), writes tests, generates mocks, and commits in the worktree. Do NOT write code yourself.
5. The coder writes **table-driven** tests for EVERY acceptance criterion and makes the gates green. If it reports it cannot reach green, send back the specific gap (or, only if your plan was wrong, revise the plan and re-delegate) — do NOT hand-edit code.
6. The coder MUST report the actual total coverage number from `make cover` (the `total: … NN.N%` line), not just "green". VERIFY it yourself: coverage ≥ 80% AND `test`/`lint` green (it also fixes any `typos`/`spell` issues). If the number is missing, unparseable, or < 80% → send the coder back for the specific gap; do NOT trust a bare "green". Then print `[IMPLEMENT] #<N> green cover=<pct>%` and keep its full test/cover/lint output for the PR.
7. SELF-REVIEW: run `git -C ../<slug> diff origin/main...HEAD` (Bash — the coder committed in the worktree, so target it explicitly and diff against `origin/main`, not a stale local `main`) to capture the coder's diff, then delegate that diff + the issue's acceptance criteria to BOTH the `code-reviewer` and the `security-reviewer` subagents (fresh context; they review what you pass in — they do NOT fetch it). Fix what they find **by delegating the changes back to the `coder` subagent** (it re-runs its green checks) — do NOT hand-edit code. GATE: if either reviewer reports a blocker that the coder cannot fully resolve (after one re-delegation), do NOT open a PR — apply `needs-human` (`mcp__github__update_issue`), comment the unresolved blocker, print `[IMPLEMENT] #<N> blocked -> needs-human`, and exit. Otherwise print `[IMPLEMENT] #<N> self-review done`.
8. **Re-sync, then push.** First bring the branch up to date so the PR opens mergeable (`main` may have moved while the coder worked): `git -C ../<slug> fetch origin && git -C ../<slug> merge origin/main` (Bash — **merge**, NOT rebase; the push must stay force-free, force-push is denied). If the merge conflicts, delegate the resolution to the `coder` (it re-runs the gates green) — do NOT hand-edit. Then **push** the branch from the worktree: `git -C ../<slug> push -u origin feat/N-<slug>` (Bash). Then open the PR via `mcp__github__create_pull_request` (base `main`, head `feat/N-<slug>`): body with `Closes #N` + the attached test/cover/lint output, **filling the PR template AND checking its boxes** (`make lint` green, service tests green, check output attached) and **listing the affected services** — never leave the Checks section empty. **Partial-failure guard:** if `create_pull_request` fails AFTER the push succeeded, retry once; if it still fails, REMOVE yourself from the issue's assignees (`mcp__github__update_issue` #N, assignees=[]) so the issue is re-claimable instead of stuck-assigned with an orphan branch and no PR, print `CYCLE_ERROR pr-create-failed #<N>`, and exit. On success print `[PR] opened #<N> for issue #<N>` (PR number then issue number).
9. Exit. Do NOT merge — the reviewer does that.
