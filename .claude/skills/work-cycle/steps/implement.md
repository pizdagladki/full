> Board status is automated by the project's GitHub workflows (issue assigned → In Progress, PR merged → Done). Do NOT touch the Projects board from here — claiming the issue (the assignment done in `select.md`) is what moves the card.

GitHub issue/PR operations use the **GitHub MCP** tools (`mcp__github__*`, owner `pizdagladki`, repo `full`); local code + git via Bash.

1. `mcp__github__issue_read` (method=get, #N). If the acceptance criteria are unverifiable/unclear → apply `needs-human` AND remove yourself from assignees (`mcp__github__issue_write` method=update, assignees=[]) — an escalated issue must not sit "claimed" (it sticks the board card In Progress and blocks nothing useful), exit. Do NOT guess.
2. **Sync first:** `git fetch origin`. Work in a separate git worktree, branching `feat/N-<slug>` off the freshly-fetched `origin/main` (NOT a stale local `main`): `git worktree add ../<slug> -b feat/N-<slug> origin/main`.
3. Plan (brain): for a non-trivial task enter plan mode and draft a plan (WITHOUT human approval); for a trivial task a one-line plan is fine. You do NOT write code yourself — the `coder` subagent does (next steps).
4. **Delegate the implementation to the `coder` subagent** (Sonnet): pass it the absolute worktree path, the zone from the issue's "Service / area" section, the acceptance criteria, and your plan. It codes strictly within that zone per `go-backend-conventions` (+ `new-service`/`new-resource`; `frontend-conventions` for a `frontend` zone), writes tests, generates mocks, and commits in the worktree. Do NOT write code yourself.
5. The coder writes **table-driven** tests for EVERY acceptance criterion and makes the gates green. If it reports it cannot reach green, send back the specific gap (or, only if your plan was wrong, revise the plan and re-delegate) — do NOT hand-edit code.
6. The coder MUST report the actual total coverage number from `make cover` (the `total: … NN.N%` line), not just "green". VERIFY it yourself: coverage ≥ 80% AND `test`/`lint` green. If the number is missing, unparseable, or < 80% → send the coder back for the specific gap; do NOT trust a bare "green". Then print `[IMPLEMENT] #<N> green cover=<pct>%` and keep its full test/cover/lint output for the PR.
7. **Re-sync FIRST** (so the diff you self-review is the FINAL one, including the merge result — never review a pre-merge diff and then push merged code): `git -C ../<slug> fetch origin && git -C ../<slug> merge origin/main` (Bash — **merge**, NOT rebase; the push must stay force-free, force-push is denied). If the merge conflicts, delegate the resolution to the `coder` (it re-runs the gates green) — do NOT hand-edit.
8. **IN-CYCLE SELF-REVIEW** (this is the objective gate; it runs HERE, warm, to avoid a separate cold review cycle — but the VERDICT is rendered by a FRESH subagent, never by you, because you wrote this code and are biased toward shipping). Capture the final diff: `git -C ../<slug> diff origin/main...HEAD` (Bash). Then:
   - **Panel** (fresh subagent windows; you attach the diff + adjacent files + VERBATIM criteria + the source spec doc path — they do NOT fetch): ALWAYS `code-reviewer` (correctness + criteria + conventions + baseline security) and `criteria-auditor` (criterion→named-test mapping + criteria-vs-spec audit); ALSO `security-reviewer` ONLY IF a changed path matches `auth|oauth|session|token|payment|stripe|storage|secret`.
   - **Verdict** — delegate to the `adjudicator` subagent. Pass it ONLY the diff, the verbatim criteria, and the panel's raw findings — NOT your plan or narrative (that contamination is exactly what the fresh window removes). It returns apply/dismiss per finding + the dismissed-findings list + GOOD/BAD; `criteria-auditor`'s `UNTESTED` is a mechanical BAD it cannot waive.
   - **BAD** → delegate the applied fixes back to the `coder` (it re-greens the gates), then re-run the panel + adjudicator on the new diff. If still BAD after ONE re-delegation → apply `needs-human` AND remove yourself from assignees (`mcp__github__issue_write` method=update, assignees=[] — an escalated issue must not sit "claimed"), comment the unresolved blocker (include the exact fix if the adjudicator named one — a human or a later re-approved cycle acts on it), print `[IMPLEMENT] #<N> blocked -> needs-human`, exit WITHOUT opening a PR.
   - **GOOD** → print `[IMPLEMENT] #<N> self-review GOOD`; keep the adjudicator's dismissed-findings list to post as one audit comment once the PR exists (step 9).
9. **Push, open PR, ARM.** Push the branch: `git -C ../<slug> push -u origin feat/N-<slug>` (Bash). Open the PR via `mcp__github__create_pull_request` (base `main`, head `feat/N-<slug>`): body with `Closes #N` + the attached test/cover/lint output, **filling the PR template AND checking its boxes** (`make lint` green, service tests green, check output attached) and **listing the affected services** — never leave the Checks section empty. **Partial-failure guard:** if `create_pull_request` fails AFTER the push succeeded, retry once; if it still fails, REMOVE yourself from the issue's assignees (`mcp__github__issue_write` method=update #N, assignees=[]) so the issue is re-claimable instead of stuck-assigned with an orphan branch and no PR, print `CYCLE_ERROR pr-create-failed #<N>`, and exit. On success, post the adjudicator's dismissed-findings list (if any) as one audit comment (`mcp__github__add_issue_comment` #N). The self-review already passed, so **ARM via the `reviewed-armed` label — you do NOT run any merge command** (the harness classifier semantically denies merge/auto-merge commands on your own PR as [Self-Approval], regardless of syntax — gh CLI and the GraphQL mutation both get blocked; labels via MCP are never blocked, and the merge mechanics live in repo automation): add the **`reviewed-armed`** label (`mcp__github__issue_write` method=update #N). The PR now merges on green CI. Print `[PR] opened+armed #<N> for issue #<N>`.
