SINGLE-AGENT MODE — read this before the cascade. The fleet now runs as ONE agent on ONE account (one subscription can't sustain async cross-account workers). There is no second account, so you **review and merge your OWN PRs** — the old "not the author / not the most-recent-pusher" exclusions are gone. Crucially, GitHub forbids approving your own PR, so review.md fires **no `APPROVE` event**: `reviewDecision` therefore never reaches `APPROVED` and is a DEAD signal — do NOT key any decision on it. The "review is done, this PR is en route to merge" signal is the label **`reviewed-armed`** (set by review.md when it arms auto-merge; cleared by any new push — address.md/unblock.md). The binding merge gate is now CI (required checks) + resolved threads + up-to-date branch; the fresh-process self-review is what decides whether to arm. (See fleet-flow.md §Гейт.)

2. CI-RECOVERY. An open PR that already passed self-review (label `reviewed-armed`, so auto-merge is armed and it is excluded from REVIEW) but whose CURRENT `head.sha` has a REQUIRED CI check with conclusion `failure`/`cancelled`/`timed_out` (read `gh pr checks <N>` or `gh pr view <N> --json statusCheckRollup`). Auto-merge can NEVER fire on a red required check, so without this clause the PR is a permanent zombie no other category catches. → strip `reviewed-armed` AND **disarm auto-merge** (`gh pr merge <N> --disable-auto`, Bash — so the coming fix is re-reviewed, not auto-merged on its first green CI), apply `needs-work` (`mcp__github__issue_write` method=update), post a comment naming the failed job(s) and their first failures, then `[PICKED] type=address #<N>` (work type = address). (A not-yet-reviewed PR carries no `reviewed-armed`; red CI on it is handled by REVIEW→BAD below, not here.)

   > INVARIANT (holds across all steps): the label `reviewed-armed` is present ⟺ auto-merge is armed. Every point that strips the label (CI-RECOVERY here, review.md BAD, unblock.md conflict-resolved) MUST disarm auto-merge, and the only point that adds the label (review.md GOOD) is the only one that arms. This is what keeps a rejected/failing/rebased PR from silently merging an unreviewed diff on green CI.

3. UNBLOCK. An open PR that GitHub will never merge on its own because its branch is no longer mergeable. This un-sticks an armed PR that falls behind `main` after another PR of the batch merges, and re-syncs a PR that conflicts with `main`. Consider only PRs NOT labeled `needs-work` (those belong to CHANGES → address.md). For each, read `gh pr view <N> --json mergeStateStatus,mergeable,autoMergeRequest,headRefName,isDraft` (Bash). It matches when ANY of:
   - `mergeable == CONFLICTING` (state `DIRTY`) — a real conflict with `main`, at ANY state; OR
   - `mergeStateStatus == BEHIND` (and `mergeable != CONFLICTING`) — behind `main`: the branch needs `main` merged in before auto-merge can fire. Mechanical re-sync, NOT a review; OR
   - `reviewed-armed` label present AND `mergeStateStatus` is `CLEAN`/`UNSTABLE`/`BLOCKED` AND `autoMergeRequest == null` — self-reviewed but auto-merge was never actually armed (re-arm it; harmless while `BLOCKED` — fires once the block clears).
   Skip drafts; treat `UNKNOWN` as "not yet — let GitHub finish computing" (re-read once, else skip). It is your own PR — this is mechanical re-sync, not review.
   - Claim it **exclusively** (assign + `🤖 [CLAIM]` stamp + 1–3 s + re-read + sole-you check). Lost the race → yield, restart selection.
   - Otherwise → `[PICKED] type=unblock #<N>`; work type = unblock, PR = #N.
