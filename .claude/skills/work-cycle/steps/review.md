Self-review PR #N (single-agent mode — it is YOUR OWN PR; there is no second account). You claimed it by assigning yourself. Print `[REVIEW] #<N> start`.

The FIRST review of a freshly-implemented PR happens in-cycle in implement.md (it arms before opening). This cycle therefore handles the rest: **re-review after `address` fixed a `needs-work` PR**, **legacy In-Review PRs** from the cross-account cutover, and **late-Copilot threads** on an already-armed PR. Its objectivity comes from ONE thing you must protect: **you are a fresh `claude -p` process with ZERO memory of writing this code** — reconstruct everything from GitHub and judge the diff as a stranger's. **You (the orchestrator) run on Sonnet for cost; the VERDICT therefore must come from the opus `adjudicator` subagent, NOT from you — keep every judgment call on Opus.** You only reconstruct context, run the panel, drive the `adjudicator`, and execute its decision (resolve threads, arm / needs-work). Do NOT rationalize toward merging to "finish the cycle". The load-bearing objectivity lives in two MODEL-INDEPENDENT signals (CI + the criterion→test mapping); the LLM panel is a context-decorrelated second look that catches presentation-level defects but shares the author's blind spots — never treat its silence as proof of correctness. (See fleet-flow.md §Гейт.)

1. **Sync first:** `git fetch origin`, then confirm you are reviewing the **latest pushed head** — `mcp__github__pull_request_read` (method=get, #N), note `head.sha`; if it changed since you claimed, re-read. Fetch the diff via `mcp__github__pull_request_read` (method=get_diff, #N). Read the linked issue's acceptance criteria via `mcp__github__issue_read` (method=get) AND note the **source spec doc** the issue records in its Context (e.g. `docs/specs/…`) — the criteria auditor needs it.
   - **THREADS-ONLY SHORT-CIRCUIT:** if the PR is already `reviewed-armed` AND there are **no new commits** since it was armed (only unresolved review threads landed — typically a late Copilot comment), do NOT re-run the panel. Gather the threads (step 3) and adjudicate+resolve them (the rules in step 5). If any thread is a real `apply` defect → take the BAD path (strip `reviewed-armed` + disarm auto-merge + `needs-work`). Otherwise verify auto-merge is still armed (re-arm if it dropped), keep `reviewed-armed`, print `[REVIEW] #<N> threads-resolved (still armed)` and exit. This keeps late-Copilot cheap (no panel).
2. **CI status (per-job, not the aggregate):** `mcp__github__pull_request_read` (method=get_check_runs, #N) (fallback `gh pr checks <N>`). Verify each expected job (`lint`, `typecheck`, `test`, `build`, `frontend`) on the CURRENT `head.sha`. Classify: any expected job with conclusion `failure`/`cancelled`/`timed_out`/stale → **CI-RED** (a hard BAD). All `success` → CI-GREEN. Some still running and none failed → CI-PENDING (OK to arm over — auto-merge waits for green; a later red is caught by select.md CI-RECOVERY).
3. **Copilot comments (advisory — must be adjudicated AND resolved):** list resolvable threads with IDs via `mcp__github__pull_request_read` (method=get_review_comments, #N) or GraphQL:
   `gh api graphql -f query='query($o:String!,$r:String!,$n:Int!){repository(owner:$o,name:$r){pullRequest(number:$n){reviewThreads(first:100){nodes{id isResolved comments(first:1){nodes{author{login} body path line}}}}}}}' -F o=pizdagladki -F r=full -F n=<N>`
   Separate out threads authored by the **Copilot reviewer bot** (type `Bot`, login like `copilot-pull-request-reviewer`; confirm on a real PR). Collect each Copilot thread's `id` + body + file/line. If Copilot's review isn't present yet → `[COPILOT] none yet` and proceed — NEVER block waiting for Copilot.
4. **Delegate the objective self-review to a calibrated panel** (fresh context; each reviewer reviews what you pass in — they do NOT fetch the diff). To keep the file-read cost bounded, YOU (cheaply, once) attach to each prompt the specific adjacent files/functions the diff touches; do NOT let three agents crawl the tree independently.
   - ALWAYS — `code-reviewer` (merged correctness + criteria-conformance + conventions + baseline security): pass the diff, the attached adjacent files, the PR metadata, the VERBATIM acceptance criteria, and the Copilot comments to classify `apply`/`dismiss`.
   - ALWAYS — `criteria-auditor` (it has Read access for the test files): pass the diff, the criteria, and the source spec doc path. It returns (a) per criterion, the NAMED test that fails on violation — or `UNTESTED` if none; and (b) requirements present in the spec doc but ABSENT from the criteria.
   - ONLY IF the diff touches a sensitive path (any changed file path matching `auth|oauth|session|token|payment|stripe|storage|secret`) — ALSO `security-reviewer` for a deeper pass. Skip it otherwise (the `code-reviewer` already did the baseline security checklist) to save tokens.
   - **Re-review (the PR was `needs-work`/CHANGES before this round):** pass the panel the NEW diff and re-derive the criterion→test mapping FROM SCRATCH. Pass only the raw FACTS of what the resolved threads asked for — do NOT hand the reviewers the prior round's prose verdict (it anchors them into rubber-stamping "addressed").
5. **Delegate the VERDICT to the `adjudicator` subagent** (opus) — do NOT adjudicate yourself (you are Sonnet; keep judgment on Opus). Pass it ONLY the diff, the verbatim criteria, and the panel's raw findings (code-reviewer + criteria-auditor [+ security-reviewer], plus the list of Copilot comments) — NOT your own framing. It returns: apply/dismiss per finding (INCLUDING per-Copilot-comment), the verbatim dismissed-findings list, and the overall GOOD/BAD. It enforces these rules (you neither re-judge nor override them):
   - **MECHANICAL, non-overridable:** any criterion the auditor marked `UNTESTED` → BAD.
   - **Spec-vs-criteria gaps:** a correctness/security requirement in the spec but missing from the criteria → BAD → needs-human (criteria incomplete; only a human owns `owner-agreed` completeness); a nice-to-have gap → just a comment.
   - **Open-ended findings:** `apply` ONLY when tied to a SPECIFIC line AND a concrete failing input/scenario or a violated criterion; else `dismiss`. A Copilot comment alone never fails the PR without a real defect.
   Then YOU (parent — you hold MCP/Bash, the adjudicator does not) carry out its decision, mechanically:
   - **Resolve EVERY Copilot thread** in BOTH verdict paths (resolution ≠ application, or require-conversation-resolution silently blocks merge): `apply` → reply it's folded in, then resolve; `dismiss` → post the adjudicator's one-line reason, then resolve. Resolve via `mcp__github__pull_request_review_write` (method=resolve_thread, threadId=<id>) or `gh api graphql -f query='mutation($t:ID!){resolveReviewThread(input:{threadId:$t}){thread{isResolved}}}' -F t=<threadId>`. Print `[COPILOT] adjudicated apply=<a> dismiss=<d> resolved=<r>`.
   - **Post the adjudicator's dismissed-findings list verbatim** as one audit PR comment (`mcp__github__add_issue_comment` #N) so every wave-through is auditable.
6. **Act on the adjudicator's verdict — you EXECUTE, you do NOT re-judge it.** (For reference, GOOD means: no applied blocker AND every criterion test-backed (no `UNTESTED`) AND no correctness/security spec-gap AND CI not CI-RED — but the adjudicator already decided this; just carry it out.)
   - **GOOD** → Print `[REVIEW] #<N> verdict=GOOD`.
     - **Mergeability gate FIRST** — only arm an up-to-date, conflict-free branch (arming a `BEHIND`/`DIRTY` branch is wasted): `gh pr view <N> --json mergeStateStatus,mergeable`. If `mergeable == CONFLICTING` (`DIRTY`) OR `mergeStateStatus == BEHIND` → do NOT arm this cycle; the code is fine, this is a merge-readiness defer. RELEASE your claim (remove yourself from assignees, `mcp__github__issue_write` method=update #N) so UNBLOCK re-syncs the branch; a fresh review then arms the up-to-date head. Print `[REVIEW] #<N> good-but-unmergeable state=<BEHIND|DIRTY> -> unblock` and exit. (`UNKNOWN` → re-read once; still unknown → defer the same way.)

Read the issue number from the PR body's `Closes #N`, check it, and if still open close it (`mcp__github__issue_write` method=update, state=closed, state_reason=completed) — the board's built-in "closed → Done" workflow then moves the card.
